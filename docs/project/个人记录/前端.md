# 前端

这份文档用于记录我对 SysY 编译器前端部分的个人理解。它不是严格的 API 文档，而是偏向面试复盘和自我梳理：我会把自己的理解写下来，同时补充容易说错的概念、项目中的实际实现方式，以及面试时可以如何表达。

## 1. 前端整体理解

当前编译器前端使用 Flex/Bison 完成源码到 AST 的转换。

更准确地说：

- Flex 负责词法分析，也就是 scanning 源文件，把字符流切分成 token。
- Bison 负责语法分析，也就是 parsing token stream，根据语法规则规约出 AST。
- Parser action 在规约过程中直接构造 AST 节点。
- AST 之后交给 lowering 阶段，由 `RewindIRBuilder` 转换为自定义 Rewind IR。

整体流程可以概括为：

```text
SysY source
  -> Flex lexer
  -> token stream
  -> Bison parser
  -> AST
  -> AST dump 或 AST lowering
```

这里需要注意一个说法修正：不是“Flex 生成 token 流之后传递给词法分析阶段”，而是 Flex 本身就是词法分析阶段。Flex 生成的 `yylex()` 函数在 Bison parser 需要下一个 token 时被调用，返回 token 类型，并通过 `yylval` 传递 token 附带的语义值，例如标识符字符串和整数值。



## 2. Flex 词法分析

项目中的词法文件是 `src/front_end/sysy.l`。它主要做几件事：

- 忽略空白符、行注释和块注释。
- 识别关键字，例如 `int`、`void`、`const`、`if`、`else`、`while`、`return`、`break`、`continue`。
- 识别运算符和分隔符，例如 `+`、`-`、`*`、`/`、`%`、`==`、`!=`、`<=`、`>=`、`&&`、`||`。
- 识别标识符。
- 识别整数常量，包括十进制、八进制和十六进制。

词法分析阶段只负责把源代码切成 token，不负责判断变量是否定义、表达式类型是否正确、数组维度是否合法。这些问题属于语法分析之后的语义检查和 lowering 阶段。

一个典型例子是：

```c
int x = a + 3;
```

Flex 只会识别出类似：

```text
INT IDENT '=' IDENT ADD INT_CONST ';'
```

它不会判断 `a` 是否已经定义，也不会判断 `a + 3` 的类型是否符合赋值目标。这些都交给后续阶段。



## 3. Bison 语法分析

项目中的语法文件是 `src/front_end/sysy.y`。它定义了 SysY 的语法产生式，并在每个 grammar action 中构造对应 AST 节点。

Bison 的语法描述形式接近 BNF。很多 EBNF 中的 `{...}`、`[...]` 在 Bison 中通常用递归产生式和 `%empty` 表达。例如：

- 列表结构用 `XXXList : XXX | XXXList XXX`。
- 可选结构用 `%empty` 或空指针表示。
- 多维数组维度用 `DimDeclList` 表示多个 `[ConstExp]`。

关于 Bison 使用的分析算法：当前 `sysy.y` 没有启用 `%glr-parser`，也没有配置 `%define lr.type`，因此可以按 Bison 默认的 LALR(1) parser 来介绍。更严谨的说法是：

> 项目使用 Bison 生成 LALR(1) 风格的 parser，通过语法改写处理表达式优先级、dangling else 等语法问题，并在规约动作中构造 AST。



## 4. 表达式优先级如何处理

表达式语法没有依赖大量 `%left`、`%right` 优先级声明，而是通过分层非终结符自然表达优先级：

```text
Exp
  -> LOrExp
  -> LAndExp
  -> EqExp
  -> RelExp
  -> AddExp
  -> MulExp
  -> UnaryExp
  -> PrimaryExp
```

这种写法的好处是结构清晰，每一层 AST 节点都对应一种优先级：

- `LOrExp` 处理 `||`。
- `LAndExp` 处理 `&&`。
- `EqExp` 处理 `==` 和 `!=`。
- `RelExp` 处理 `<`、`>`、`<=`、`>=`。
- `AddExp` 处理 `+` 和 `-`。
- `MulExp` 处理 `*`、`/`、`%`。
- `UnaryExp` 处理一元运算和函数调用。
- `PrimaryExp` 处理数字、括号表达式和左值。

这种结构也方便 lowering：不同表达式节点可以递归 lower 成对应 IR value。



## 5. 语法冲突和 dangling else

一个典型难点是 dangling else，也就是：

```c
if (a)
  if (b)
    x = 1;
  else
    x = 2;
```

这里的 `else` 应该绑定到最近的尚未匹配 `else` 的 `if`，也就是内层 `if (b)`。

如果使用朴素文法：

```text
Stmt -> IF '(' Exp ')' Stmt
      | IF '(' Exp ')' Stmt ELSE Stmt
```

parser 在看到 `else` 时会遇到经典的 `shift/reduce conflict`：它既可以把前面的 `if (...) Stmt` 规约成一个完整语句，也可以继续 shift `else`，让 `else` 绑定到这个 `if`。一般语言规则希望 shift，让 `else` 绑定最近的 `if`。

本项目没有单纯依赖冲突默认规则，而是把语句拆成：

```text
MatchedStmt
UnMatchedStmt
```

含义是：

- `MatchedStmt` 表示所有 `if` 都已经匹配好了 `else` 的语句。
- `UnMatchedStmt` 表示还存在一个没有匹配 `else` 的 `if`。

当前 `grammar` 中：

```text
Stmt
  -> MatchedStmt
  -> UnMatchedStmt
```

`if (...) MatchedStmt else MatchedStmt` 是 matched；`if (...) Stmt` 是 unmatched。

这样就把“else 绑定最近 if”的规则编码进文法结构里，避免把语义交给冲突默认处理。

面试时可以这样说：

> dangling else 的本质是朴素 if 文法存在移进/规约冲突。我这里通过 MatchedStmt 和 UnMatchedStmt 改写文法，让 parser 从语法结构上区分已经匹配 else 的语句和还没匹配 else 的语句，从而保证 else 绑定到最近的 if。



## 6. 加入 while 后为什么也要小心

`while` 本身没有 `else`，看起来不像 dangling else 的来源，但它的 body 是一个 `Stmt`，而 `Stmt` 里面可能包含未匹配的 `if`。所以加入 `while` 之后，不能只把 `while '(' Exp ')' Stmt` 随便塞进某一个产生式里，否则可能破坏 matched/unmatched 的分类。

当前 grammar 里的处理方式是：

```text
MatchedStmt
  -> WHILE '(' Exp ')' MatchedStmt

UnMatchedStmt
  -> WHILE '(' Exp ')' UnMatchedStmt
```

这个设计表达了一个关键点：

- 如果 `while` 的 body 是 matched，那么整个 `while` 也是 matched。
- 如果 `while` 的 body 是 unmatched，那么整个 `while` 也是 unmatched。

也就是说，`while` 会传播 body 的 matched/unmatched 状态。

例如：

```c
if (a)
  while (b)
    if (c)
      x = 1;
    else
      x = 2;
```

这里 `else` 应该绑定到 `if (c)`，不是外层 `if (a)`。因为 `while` 的 body 可以是一个完整 matched 的 `if-else`，所以整个 `while` 可以成为 `MatchedStmt`，外层 `if (a)` 则仍然是一个没有 `else` 的 `UnMatchedStmt`。

再看：

```c
if (a)
  while (b)
    if (c)
      x = 1;
else
  x = 2;
```

这里 `while` 的 body 中有一个未匹配 `else` 的 `if (c)`。这种情况下，`while (...) UnMatchedStmt` 整体也要被视为 `UnMatchedStmt`，这样后面的 `else` 会优先绑定到最近的未匹配 `if`，而不是错误地绑定到外层 `if (a)`。

所以 while 的难点不是它自己有冲突，而是它包裹的语句可能携带 dangling else 状态。把 while 同时放入 `MatchedStmt` 和 `UnMatchedStmt`，本质是在维护“未匹配 if”状态的传播。



## 7. AST 设计

AST 定义主要在 `include/front_end/ast.h`，Dump 实现在 `src/front_end/ast.cpp`。

整体设计可以概括为：

- 所有 AST 节点继承 `BaseAST`。
- `BaseAST` 提供虚析构函数和 `Dump` 接口。
- 节点之间用 `std::unique_ptr<BaseAST>` 表达树形所有权。
- 列表结构用 `std::vector<std::unique_ptr<BaseAST>>` 表达。
- 同一个语法节点的多种形态用 `std::variant` 表达。

例如：

- `CompUnitAST` 保存全局 `Decl` 和 `FuncDef` 列表。
- `FuncDefAST` 保存函数返回类型、函数名、形参列表和函数体 block。
- `DeclAST` 统一包装 `ConstDeclAST` 和 `VarDeclAST`。
- `StmtAST` 用 `variant` 表达 `Return`、`Assign`、`Block`、`Exp`、`SelectStmt`、`LoopStmt`、`LoopControlStmt`。
- `UnaryExpAST` 用 `variant` 表达 primary、一元运算和函数调用。
- `PrimaryExpAST` 用 `variant` 表达数字、括号表达式和左值。
- `LValAST` 保存变量名和多维数组下标列表。

AST 的目标不是提前做优化，而是保留源程序的结构，给后续 lowering 足够的信息。比如数组访问：

```c
a[i][j]
```

在 AST 中会保留为：

```text
LValAST
  ident = "a"
  indices = [i, j]
```

到了 lowering 阶段，再根据符号表判断 `a` 是局部数组、全局数组还是数组形参，并生成对应的 `getptr` 或 `getelemptr`。



## 8. 前端和语义分析的边界

当前项目的语义检查并没有独立成一个完整 semantic pass，而是和 AST 到 IR 的 lowering 过程结合在一起。也就是说，parser 主要负责构造 AST，lowering 阶段会顺手完成很多语义检查。

例如：

- 查找变量、常量和函数是否定义。
- 检查函数调用的参数数量和类型。
- 检查 `return` 是否符合函数返回类型。
- 检查数组是否被正确索引。
- 检查 `ConstExp` 是否可以编译期求值。
- 处理数组初始化、数组形参 decay、全局变量初始化等。

这种做法的好处是实现直接，适合项目早期快速打通完整链路；缺点是 lowering 代码会变大，语义检查和 IR 生成容易耦合。后续重构方向是把符号表、常量求值、数组初始化、表达式 lowering、语句 lowering 逐步拆开，让前端、语义和 IR 生成边界更清楚。



## 9. 面试时可以如何讲前端

可以用下面这段话作为简洁回答：

> 前端使用 Flex/Bison 实现。Flex 负责扫描 SysY 源码，忽略空白和注释，识别关键字、标识符、整数常量和运算符，并通过 `yylex()` 把 token 提供给 Bison。Bison 默认生成 LALR(1) 风格 parser，我在 `sysy.y` 中用递归产生式描述 SysY 语法，并在规约动作里构造 AST。表达式优先级通过 `LOrExp` 到 `PrimaryExp` 的分层非终结符表达；dangling else 通过 `MatchedStmt` 和 `UnMatchedStmt` 改写文法解决；加入 while 后，需要让 while 根据 body 的 matched/unmatched 状态分别出现在两个产生式中，保证 else 仍然绑定最近的 if。AST 设计上，所有节点继承 `BaseAST`，用 `unique_ptr` 管理树形所有权，用 `variant` 表达节点的不同形态。真正的符号检查、常量求值、数组初始化和函数调用类型检查主要发生在 AST 到 IR 的 lowering 阶段。



## 10. 容易被追问的问题

**问题：Bison 是不是直接支持 EBNF？**

Bison 的产生式更接近 BNF。EBNF 里的可选项和重复项，通常需要用 `%empty` 和递归 list 产生式手动表达。

**问题：Bison 默认是什么 parser？**

当前项目没有启用 `%glr-parser`，也没有配置 `%define lr.type`，所以可以按 Bison 默认 LALR(1) parser 来讲。

**问题：dangling else 是什么冲突？**

朴素 if 文法会产生 shift/reduce conflict。看到 `else` 时，parser 可以选择规约前面的 `if (...) Stmt`，也可以选择继续移进 `else`。语言规则要求 `else` 绑定最近的 `if`。

**问题：为什么 while 也要放进 MatchedStmt 和 UnMatchedStmt？**

因为 while 的 body 是一个 statement，而这个 body 可能包含未匹配 else 的 if。while 需要传播 body 的 matched/unmatched 状态，否则会破坏 dangling else 的绑定规则。

**问题：为什么语义检查不直接放在 parser 中？**

Parser 更适合做语法结构构造。符号表、类型、数组 decay、函数调用匹配等需要上下文信息，放在 lowering 阶段统一处理更方便，也能直接生成对应 IR。

**问题：AST 为什么用 `variant`？**

同一个语法节点有多种形态，例如 `Stmt` 可以是 return、assign、block、if、while。用 `variant` 比手写 kind 字段和裸指针更类型安全，也更贴近 C++17 风格。
