# 中间表示

这份文档记录我对当前 SysY 编译器中间表示部分的理解。中间表示是前端 AST 和后端 RISC-V 之间的边界层：前端把源语言中的表达式、声明、控制流、数组访问、函数调用等语义 lowering 成 IR；后端只需要面对更规则的 `Module / Function / BasicBlock / Value` 结构。

## 1. 整体设计

当前自定义 IR 可以按四个核心概念理解：

- `IRModule`：整个编译单元的 IR 容器，也是 AST lowering 的最终产物。
- `IRFunction`：函数级 IR，包含函数类型、形参、基本块，以及是否为外部声明。
- `IRBasicBlock`：基本块，保存一段顺序执行的 IR 指令。
- `IRValue`：IR 的统一值基类，常量、聚合初始化值、函数参数引用、以及各种指令都挂在这个体系下。

这个设计接近 LLVM IR 的核心思想：把“值”和“产生值的指令”统一到同一套对象模型中。这样表达式 lowering 后可以直接返回一个 `IRValue*`，后续指令如果需要使用它，只需要保存这个 value 指针即可。

需要注意的是，当前项目还不是完整 SSA 形式。它已经有 SSA 风格的临时值命名，例如 `%0`、`%1`，但局部变量仍然主要通过 `alloc / load / store` 表达。后续如果引入 `mem2reg`、`phi`、IR pass，可以在现有 Value 和 BasicBlock 模型上继续演进。



## 2. IRModule

`IRModule` 表示一个完整编译单元，里面主要包含全局对象和函数：

- `funcs_`：模块中所有函数的非拥有指针列表，便于遍历输出或交给后端处理。
- `global_values_`：全局变量、全局数组等全局 value 的非拥有指针列表。
- `func_storage_ / bb_storage_ / value_storage_`：使用 `vector<unique_ptr<...>>` 真正拥有所有 IR 对象资源。

这种设计把“对象生命周期管理”和“IR 图结构引用”分开了：`IRModule` 统一持有资源，其他结构之间只保存裸指针作为引用关系。这样可以避免到处手动 delete，也能让 Function、BasicBlock、Instruction 之间互相引用时保持简单。

`IRModule` 同时提供创建接口，例如：

- `make_function`：创建函数对象，并放入模块函数列表。
- `make_basic_block`：创建基本块对象，之后再追加到某个函数中。
- `make_value<T>`：创建某种 IR value 或 instruction。
- `append_basic_block / append_value / append_global_value / append_param`：维护 Function、BasicBlock、Module 内部的非拥有索引关系。

所以可以这样概括：AST lowering 的最终结果就是一个 `IRModule`，它既是 IR 的根节点，也是 IR 对象的资源拥有者。



## 3. IRFunction

`IRFunction` 对应一个函数定义或函数声明。它主要包含：

- `type_`：函数类型，类型中包含参数类型列表和返回类型。
- `name_`：函数名，例如 `@main`。
- `params_`：函数形参 value，当前用 `IRFuncArgRef` 表示。
- `basic_blocks_`：函数体中的基本块列表。
- `is_declaration_`：区分函数声明和函数定义，例如 SysY 运行库函数只需要声明，不在当前源文件中定义。

函数声明和函数定义在 IR 中都用 `IRFunction` 表示，这是比较自然的设计。声明函数只有类型、名字和参数信息，没有函数体；定义函数则额外拥有基本块和指令。

形参本身也是 `IRValue`。在 lowering 函数体时，当前实现会为形参创建局部 `alloc`，然后把形参值 `store` 进去。这样后续代码访问形参时，可以和普通局部变量一样走 `load / store` 路径，降低前端 lowering 的复杂度。



## 4. IRBasicBlock

`IRBasicBlock` 表示一段顺序执行的指令序列，当前保存为 `std::vector<IRValue*> insts_`。虽然容器元素类型是 `IRValue*`，但语义上这里存放的是 instruction value，因为 `IRInstruction` 继承自 `IRValue`。

基本块需要满足两个重要性质：

- 只有一个入口点：控制流只能跳到基本块开头，不能跳到基本块中间。
- 只有一个出口点：控制流转移只能由基本块最后一条 terminator 指令完成，例如 `ret`、`jump`、`branch`。

当前代码主要通过 lowering 过程维护这个约束。例如 `FuncContext::terminate_with_return / terminate_with_jump / terminate_with_branch` 会创建终结指令，并清空当前 basic block，防止继续往已经结束的块里插入指令。不过严格来说，项目后续最好补一个 `IRVerifier`，专门检查每个基本块是否正好以 terminator 结束，以及 terminator 后面是否没有普通指令。



## 5. IRValue 和 IRInstruction

`IRValue` 是 IR 的核心。它至少包含三个基本信息：

- `name_`：IR 名字，例如 `%0`、`@a_1`，常量可以没有名字。
- `kind_`：value 种类，例如`整数常量、load、store、call、alloc、getelemptr` 等。
- `type_`：value 的 IR 类型，例如 `i32`、`unit`、pointer、array、function。

这里最容易误解的一点是：不是 `IRValue` “包含” `IRInstruction`，而是 `IRInstruction` 继承自 `IRValue`。也就是说，指令是 value 的一种。

这样设计的原因是：很多指令本身会产生一个结果，而这个结果可以被其他指令继续使用。例如：

```text
%0 = load @x
%1 = add %0, 1
ret %1
```

在这种情况下，`load` 指令既是一条指令，也是后续 `add` 可以引用的 value。`binary`、`load`、非 `unit` 的 `call`、`getptr/getelemptr` 等都属于这种“会产生可用结果”的指令。

但并不是所有 instruction 都有普通意义上的返回值。`store`、`jump`、`branch`、`ret` 更像副作用或控制流指令，它们同样放在 `IRInstruction` 体系中，但类型通常是 `unit` 或者不作为表达式值继续使用。

因此，更准确的说法是：当前 IR 采用“值和指令统一建模”的方式，所有可以被引用的结果、常量、函数参数、以及指令节点都通过 `IRValue` 体系表达。



## 6. 类型系统

IR 类型由 `IRType` 体系表达，当前主要有：

- `i32`：SysY 中的整数类型。
- `unit`：类似 void，用于没有返回值的函数或没有普通结果的指令。
- `pointer`：指针类型，例如 `*i32`、`*[i32, 4]`。
- `array`：数组类型，例如 `[i32, 4]`、`[[i32, 4], 3]`。
- `function`：函数类型，包含参数类型列表和返回类型。

类型对象由 `IRTypeContext` 统一创建和复用。这样做的好处是，同一种类型在项目中可以共享同一个类型对象，类型比较也更简单。同时，`IRTypeContext` 里已经有 `getTypeSize / getTypeAlign`，这为后端栈帧布局和未来 DataLayout 抽象留下了接口。



## 7. AST Lowering 到 IR

IR 生成的主流程可以概括为：

```text
CompUnitAST
  -> RewindIRBuilder::build
  -> IRModule
  -> global values + function declarations + function definitions
  -> basic blocks + instructions
```

当前 lowering 代码大量使用 `std::visit` 搭配 `std::variant` 来处理 AST 节点中的不同分支。比如声明可能是常量声明或变量声明，表达式可能是一元表达式、二元表达式、LVal、函数调用等，使用 `std::visit` 可以把“当前 AST 节点具体是哪一种情况”分发到对应的 lowering 逻辑。

`RewindIRBuilder` 在模块层面维护 `module_symbols_`，它负责全局常量、全局变量、全局数组和函数名。函数体 lowering 时，会额外创建 `FuncContext`，用来保存当前函数内的可变状态。

当前 lowering 大致分两阶段处理函数：

- 第一阶段先声明函数，把函数名、函数类型、形参 value 放进 `IRModule` 和全局函数符号表。
- 第二阶段再 lowering 函数体，创建 entry block、局部作用域、形参 alloc/store，以及函数体中的语句和表达式。

这种先声明再定义的方式很重要，因为 SysY 里可能在函数体中调用其他函数。只要先把所有函数名和类型放入全局函数表，函数调用 lowering 时就可以通过函数名找到目标函数，并检查实参类型是否匹配。



## 8. FuncContext

`FuncContext` 是函数级 lowering 状态，它避免把所有状态都塞进 `RewindIRBuilder`。它主要保存：

- 当前 `IRModule` 引用。
- 当前正在 lowering 的 `IRFunction*`。
- 当前插入指令的 `IRBasicBlock*`。
- 当前函数内的 `SymbolTable`。
- while 循环的 `LoopTargets` 栈。
- value、alloc、basic block 的命名计数器。

`FuncContext` 中的 `LoopTargets` 用来处理 `break` 和 `continue`。进入 while 时记录当前循环的 break 目标块和 continue 目标块；遇到 `break` 或 `continue` 时，就能生成跳转到对应 basic block 的 `jump` 指令。

`ScopeGuard` 是一个 RAII 辅助结构，用来维护函数内局部作用域。进入 `{ ... }` 这样的 block 时自动 `enter_scope`，离开作用域时自动 `exit_scope`。这个设计可以减少异常或提前返回时忘记弹出作用域的问题。

命名接口也集中在 `FuncContext` 中：

- `next_percent_name()`：生成 `%0`、`%1` 这类临时 value 名。
- `next_at_name(ident)`：生成 `@a_1` 这类 alloc 或对象名。
- `next_block_name(hint)`：生成 `%entry_1`、`%while_body_1` 这类 basic block 名。

这几个命名函数让 IR 文本更稳定，也让调试输出更容易阅读。



## 9. SymbolTable 和语义检查

当前语义检查和 IR lowering 是交织在一起完成的。也就是说，编译器在遍历 AST 生成 IR 的同时，会借助 `SymbolTable` 检查变量是否定义、是否重复定义、const 是否被修改、函数是否存在、调用参数是否匹配等问题。

`SymbolTable` 主要分成两部分：

- `scopes_`：作用域栈，保存变量和常量。
- `func_table_`：函数表，保存函数名到 `IRFunction*` 的映射。

变量和常量查找使用 `lookup_value`。它会从内层作用域向外层作用域查找，所以局部变量可以遮蔽外层同名变量。函数查找使用 `lookup_function`，当前函数表是模块级的，不受局部 block scope 影响。

`LookupResult` 目前是 `std::variant<Constant, Var>`：

- `Constant`：表示编译期整数常量，例如 `const int N = 10;`，可以用于常量表达式求值。
- `Var`：表示变量或数组对象，里面保存对应的 `alloc/global_alloc` value，并通过 `is_const` 标记这个对象是否不可修改。

这里有一个重要边界：标量 const 可以直接作为编译期常量进入 `Constant`；数组即使是 const array，也不能简单当成一个编译期整数值。数组更适合以 `Var + is_const` 的方式记录其存储对象，再通过语义检查禁止对 const array 元素赋值。

项目里实际有两套符号表状态：

- `module_symbols_`：模块级符号表，记录全局常量、全局变量、全局数组和函数。
- `FuncContext::symbols_`：函数级符号表，记录形参和局部变量，并随着 block 嵌套进入/退出作用域。

`RewindIRBuilder::lookup_value` 会先查当前函数局部符号表，再查模块级符号表。这符合常见语言规则：局部名字优先，全局名字兜底。`lookup_function` 则直接查模块级函数表。



## 10. 当前设计的优点和后续改进

当前 IR 设计的优点：

- `Module / Function / BasicBlock / Value` 分层清晰，便于解释前端到后端的数据流。
- `IRValue` 统一常量、参数、指令结果，使表达式 lowering 和后端取值都比较自然。
- `IRFunctionType`、`IRPointerType`、`IRArrayType` 让数组、指针和函数调用检查有明确类型依据。
- `IRModule` 统一资源所有权，避免 IR 对象生命周期分散。
- `FuncContext` 把函数级状态从 builder 中拆出来，是后续继续重构 lowering 的良好起点。

当前仍然可以改进的地方：

- 语义检查和 IR 生成仍然耦合较重，后续可以拆出独立 Semantic Analyzer。
- BasicBlock 约束主要靠 lowering 保证，后续可以增加 `IRVerifier`。
- 当前 IR 还不是完整 SSA，后续可以引入 `phi`、`mem2reg`、CFG 分析和优化 pass。
- 数组初始化、左值地址计算、函数调用参数转换等逻辑仍然偏复杂，适合继续从 `ir_builder.cpp` 中拆分为更清晰的子模块。



## 11. 面试讲述版本

如果面试中需要快速介绍中间表示，可以这样说：

> 我在项目中设计了一层自定义 IR，作为 AST 和 RISC-V 后端之间的中间层。它的核心结构是 Module、Function、BasicBlock 和 Value。Module 是整个 IR 的根对象和资源拥有者，Function 保存函数类型、形参和基本块，BasicBlock 保存顺序执行的指令，Value 则统一表示常量、函数参数、指令结果和部分初始化值。这个设计参考了 LLVM IR 中“指令也是值”的思想，所以表达式 lowering 后可以直接返回 `IRValue*`，后续指令通过 value 引用建立数据依赖。
>
> 在 lowering 过程中，我使用 `std::variant + std::visit` 分发不同 AST 节点，并通过 `FuncContext` 保存当前函数、当前基本块、局部符号表、循环跳转目标和命名计数器。符号表使用作用域栈处理局部变量遮蔽和 block 嵌套，同时有模块级函数表支持函数声明和函数调用检查。当前语义检查和 IR 生成还在同一遍 AST 遍历中完成，后续计划拆出独立语义分析阶段，并在 IR 层补充 verifier、SSA 化和优化 pass。
